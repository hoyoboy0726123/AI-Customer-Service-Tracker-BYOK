## 1. 簡介

### 1.1 文件目的
本文件旨在詳述「AI 智慧客服問題管理系統」的產品需求、功能範圍、技術架構及成功指標。本文件將作為開發團隊（前端、後端、AI 工程師）、設計團隊 (UI/UX) 及測試團隊 (QA) 的核心參考依據，確保所有關係人對最小可行產品 (MVP) 的目標達成共識。

### 1.2 產品背景
目前客服部門在處理客戶問題時，面臨資訊極度碎片化的挑戰。問題來源包含 Line 訊息、Email、PDF 附件以及口頭交辦。由於缺乏統一的追蹤平台，導致以下問題頻發：
- **資訊散落**：客服人員需在多個視窗間切換，容易遺漏細節。
- **責任歸屬不明**：案件進入後無明確指派機制，導致「沒人負責」的案件堆積。
- **進度黑盒**：主管無法即時掌握案件的處理狀況，難以發現流程瓶頸或評估團隊工作量。
- **行政負擔**：手動輸入案件資訊與撰寫回覆信件佔用了大量核心服務時間。

### 1.3 產品願景
透過 AI 技術將非結構化的客服溝通轉化為結構化的管理數據，打造一個直觀、自動化且透明的協作平台，讓客服人員專注於解決問題而非整理資料，並賦予管理者精準決策的能力。

### 1.4 MVP 目標
1.  **數據統一化**：提供統一的入口，支援手動輸入、PDF 上傳與 Email 內容貼入。
2.  **AI 自動結構化**：利用 LLM 自動識別並填充客戶名稱、產品編號、摘要與建議結案日期。
3.  **流程可視化**：透過 Kanban (看板) 與清單報表，即時呈現案件狀態（待處理、處理中、已解決）。
4.  **責任明確化**：建立負責人指派功能，確保每個案件皆有對應的承辦人。
5.  **溝通效率化**：AI 自動生成回覆草稿，並與 Outlook 草稿匣實現一鍵連動。
6.  **管理洞察化**：提供即時統計圖表，讓主管一眼看出團隊負荷與處理效率。

## 2. 目標用戶

### 主要用戶
1.  **第一線客服人員 (CS Agent)**：負責接收、登錄、處理案件並回覆客戶。
2.  **客服主管 (CS Manager/Lead)**：負責分配資源、監控進度、排除流程瓶頸。

### 用戶特徵
- **客服人員**：
    - 每日需處理 50-100 件以上的客戶諮詢。
    - 頻繁切換通訊軟體與文書軟體。
    - 追求操作的簡便性與快速輸入。
    - 技術能力一般，偏好直觀的圖形介面。
- **客服主管**：
    - 重視數據報表與效率指標。
    - 需要在短時間內判斷哪些案件逾期或急需支援。
    - 需平衡團隊各成員的工作負荷。

## 3. 核心問題與解決方案

- **核心問題 1：溝通管道碎片化，導致案件被遺忘或重複處理。**
    - **MVP 解決方案**：建立中央集中管理平台，所有來自不同管道的訊息經由 AI 處理後統一進入系統清單，並透過強制性的狀態追蹤防止案件消失。

- **核心問題 2：手動錄入案件與填寫欄位極度耗時且易出錯。**
    - **MVP 解決方案**：引入 AI 識別模組。客服只需貼上雜亂文字或上傳文件，系統自動抓取「客戶名稱」、「產品編號」等關鍵欄位，減少 70% 的打字工作。

- **核心問題 3：主管無法即時掌握工作量分佈與處理瓶頸。**
    - **MVP 解決方案**：提供視覺化看板 (Kanban) 與即時統計儀表板 (Dashboard)，將案件進度與負責人負載轉化為直觀的圖表。

- **核心問題 4：回覆信件格式不一且撰寫緩慢。**
    - **MVP 解決方案**：AI 根據案件處理進度自動產出回覆草稿，並透過 Microsoft Graph API 將草稿直接推送到 Outlook，實現工作流的無縫接軌。

## 4. 產品功能 (MVP 範圍)

### 4.1 AI 智慧輸入與識別模組
*   **功能描述**：將雜亂的輸入源轉換為結構化的任務資料。
*   **具體子功能**：
    *   **多格式輸入介面**：提供一個大型文本輸入框（支援貼入 Line/Email 內容）以及 PDF 檔案上傳區。
    *   **AI 欄位擷取引擎**：整合 LLM (如 GPT-4o)，自動解析內容並提取：客戶名稱、產品編號、問題詳細摘要。
    *   **智慧日期建議**：AI 根據問題嚴重程度或內容中的時間關鍵字，建議「預計結案日期」。
    *   **手動修正機制**：人員可在 AI 生成後，對欄位內容進行最後的校對與手動修改。

### 4.2 案件中心 (Issue Center)
*   **功能描述**：管理所有案件的生命週期與詳細資訊。
*   **具體子功能**：
    *   **統一案件列表**：支援關鍵字搜尋、日期範圍篩選、優先級排序。
    *   **案件詳情頁**：展示問題起源、AI 提取的結構化資料、所有附件、處理紀錄。
    *   **狀態切換系統**：提供「待處理 (To-do)」、「處理中 (In Progress)」、「已解決 (Resolved)」三種核心狀態。
    *   **操作日誌**：記錄誰在什麼時間更改了狀態或負責人，確保審計追蹤 (Audit Trail)。

### 4.3 團隊指派與協作
*   **功能描述**：解決「責任歸屬」問題，確保案件有人跟進。
*   **具體子功能**：
    *   **人員管理清單**：系統維護一份客服團隊成員清單。
    *   **負責人指派功能**：支援在創建案件時指派，或在案件列表中批量轉派負責人。
    *   **個人任務視圖**：客服人員可以一鍵切換到「我的案件」，排除其他干擾。

### 4.4 視覺化管理看板 (Kanban)
*   **功能描述**：以視覺化方式呈現流程，方便主管進行資源調度。
*   **具體子功能**：
    *   **三欄式看板**：分別對應三種核心狀態。
    *   **拖拉式操作 (Drag & Drop)**：支援透過拖拉卡片快速更改案件狀態。
    *   **卡片警告標示**：若案件接近預計結案日期或已逾期，卡片會顯示顏色警示（如紅色）。

### 4.5 外部溝通整合模組 (Outlook Integration)
*   **功能描述**：加速結案後的客戶通知流程。
*   **具體子功能**：
    *   **AI 回覆建議**：根據案件的處理結果（例如：已更換零件、已軟體更新），自動撰寫專業的 Email 回覆內容。
    *   **Outlook 草稿發送**：整合 Outlook API，按下一鍵後，自動在用戶的 Outlook 帳號中生成一封帶有收件人、主旨與內文的草稿。

### 4.6 管理者統計儀表板 (Manager Dashboard)
*   **功能描述**：提供量化數據，支援管理決策。
*   **具體子功能**：
    *   **狀態分佈圓餅圖**：即時顯示目前所有案件的進度比例。
    *   **團隊負荷長條圖**：顯示每位負責人名下的待處理案件數量。
    *   **逾期案件清單**：獨立區塊列出所有超過預計結案日期且未結案的案件。

## 5. 用戶故事 (User Stories)

1.  **作為** 客服人員，**我想要** 直接將客戶在 Line 發送的雜亂訊息貼入系統，**以便** AI 能自動幫我填好客戶名稱和問題摘要，節省錄入時間。
2.  **作為** 客服人員，**我想要** 上傳客戶提供的 PDF 故障報告，**以便** 系統能自動提取其中的產品序號，減少手動輸入錯誤。
3.  **作為** 客服人員，**我想要** 在處理完問題後由系統生成信件草稿並傳送到 Outlook，**以便** 我能快速微調後發送給客戶，完成結案。
4.  **作為** 客服主管，**我想要** 在看板上看到所有案件的分佈，**以便** 我能快速發現哪些案件在「處理中」卡得太久。
5.  **作為** 客服主管，**我想要** 查看每位成員目前手上的案件量，**以便** 我在指派新案件時能做到勞逸結合。
6.  **作為** 客服主管，**我想要** 系統自動標示出已經逾期的案件，**以便** 我能介入並提供協助或向上級回報。
7.  **作為** 客服人員，**我想要** 在個人視圖中只看到指派給我的案件，**以便** 我能專注於當下的工作目標。
8.  **作為** 客服主管，**我想要** 看到「已解決」與「待處理」的週增長趨勢，**以便** 評估本週團隊的工作績效。

## 6. 非 MVP 範圍 (未來考量)

1.  **直接 API 串接**：自動從 Line Official Account 或 Gmail 提取訊息（無需手動貼入）。
2.  **多國語言支援**：支援英文、日文等介面與 AI 識別。
3.  **自動發信**：跳過 Outlook 草稿，直接由系統自動發送結案信件。
4.  **客戶門戶 (Customer Portal)**：讓客戶可以登入查看自己案件的處理進度。
5.  **進階 AI 情緒分析**：自動識別客戶訊息中的情緒（憤怒、急迫），並自動提升優先級。
6.  **行動端 App**：提供 iOS/Android App，讓人員外勤時也能更新案件。
7.  **自定義欄位**：允許主管根據不同產品線自定義案件的欄位。
8.  **權限層級控制 (RBAC)**：更細緻的角色權限管理（如：唯讀、編輯、刪除權限）。
9.  **SLA 警示通知**：透過 Slack 或 Teams 發送即時的逾期警告通知。
10. **報表定期匯出**：支援將統計圖表匯出為 PDF 或 Excel 報表。
11. **自動分配引擎**：根據人員目前的 Workload 自動輪詢 (Round Robin) 指派新案件。
12. **知識庫 (Knowledge Base)**：根據案件內容自動推薦解決方案或過往類似案例。

## 7. 技術要求

### 7.1 後端框架：Node.js (NestJS)
- **理由**：NestJS 提供強大的架構支援，易於維護與擴展，且與 AI SDK (如 OpenAI Node SDK) 整合性極佳。

### 7.2 資料庫：PostgreSQL
- **理由**：穩定的關聯式資料庫，適合處理結構化的案件數據，並支援複雜的查詢與聚合運算以提供儀表板統計。

### 7.3 資料庫 ORM：Prisma
- **理由**：提供型別安全 (Type-safety)，開發效率高，且易於進行資料庫遷移 (Migration)。

### 7.4 前端技術：React (Tailwind CSS + Shadcn UI)
- **理由**：React 生態系豐富，Shadcn UI 與 Tailwind 可以快速構建專業且具備響應式設計的企業級介面。圖表部分建議選用 **Recharts**。

### 7.5 其他整合
- **AI 模型**：OpenAI API (GPT-4o 或 GPT-4-turbo) 用於文字/PDF 解析與草稿生成。
- **檔案處理**：`pdf-parse` 或 Azure Form Recognizer 用於更深層的 PDF 解析。
- **外部連接**：Microsoft Graph API 用於 Outlook 草稿匣連動。

### 7.6 部署環境
- **建議**：容器化部署 (Docker)，主機可選擇 AWS 或 Vercel (前端) + Render/Heroku (後端)。

### 7.7 版本控制
- **建議**：使用 Git，並遵循 Git Flow 分支管理策略（main, develop, feature branches）。

## 8. 成功指標

1.  **效率提升**：單個案件從登錄到分配的平均時間減少 **50% 以上**。
2.  **資料準確度**：AI 欄位擷取的正確率（經人工微調前）達到 **85% 以上**。
3.  **用戶滿意度**：內部客服人員對系統易用性的評分（1-5 分）平均達到 **4 分以上**。
4.  **遺漏率降低**：在系統上線三個月後，未被指派或被遺忘的案件率降至 **0%**。
5.  **回覆速度**：利用 AI 草稿功能後，平均回覆客戶的時間縮短 **30%**。
6.  **功能完成度**：MVP 階段所定義的所有核心功能需在發布日 100% 準確上線。

## 9. 假設與約束

### 9.1 假設
1.  **API 穩定性**：假設 OpenAI 與 Microsoft Graph API 在開發期間保持高可用性。
2.  **資料品質**：假設用戶貼入的訊息雖雜亂，但仍包含足夠的關鍵關鍵字以供 AI 識別。
3.  **網路環境**：假設客服人員的工作環境具有穩定的網際網路連線以調用雲端 AI 服務。
4.  **權限授權**：假設公司 IT 部門允許串接 Outlook 的 OAuth 授權。
5.  **PDF 格式**：假設大部分 PDF 是數位生成的而非模糊的手寫掃描檔。
6.  **瀏覽器兼容性**：假設用戶皆使用現代瀏覽器 (Chrome, Edge, Safari)。

### 9.2 約束
1.  **時間限制**：MVP 開發週期必須在 8-12 週內完成。
2.  **預算限制**：AI API 的調用成本需控制在每個月預算範圍內，需設計 Token 使用優化機制。
3.  **隱私約束**：系統處理的客戶資料必須符合基本的資訊安全規範，不得在未經加密的情況下存儲敏感個資。
4.  **技術限制**：AI 模型單次處理的文字長度 (Context Window) 有其上限。
5.  **法規約束**：回覆草稿必須經過人工檢查，系統不應具備自動「發送」權限以規避法律風險。
